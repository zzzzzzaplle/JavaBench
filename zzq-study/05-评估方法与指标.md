# 评估方法与指标

## 一、评估流程概览

```
代码生成 → 编译检查 → 单元测试 → 计算指标
    ↓           ↓           ↓          ↓
 samples    compilation  test run   metrics
```

## 二、评估粒度

### 2.1 三种评估粒度对比

| 粒度 | 说明 | 评估方式 | 适用场景 |
|------|------|----------|----------|
| **类级别** | 一次替换一个生成的类 | 单个类 vs 标准答案 | 快速评估 |
| **测试级别** | 为每个测试替换相关类 | 多个类协同评估 | 精细评估 |
| **项目级别** | 替换所有生成的类 | 整体项目评估 | 端到端评估 |

### 2.2 类级别评估（Class-wise）

**定义**：一次只替换一个生成的类，其他类使用标准答案

**优点**：
- 评估单个类的生成质量
- 快速定位问题
- 计算资源消耗少

**缺点**：
- 忽略类间依赖
- 可能掩盖部分问题

**示例**：
```bash
python evaluation.py class-wise \
    --output output/result-PA19/gpt-3.5-turbo/single_class.json \
    output/result-PA19/gpt-3.5-turbo/samples.jsonl
```

**输出格式**：
```json
[
  {
    "task_id": "PA19/Cell.java",
    "compile_errors": 0,
    "test_result": [15, 15],
    "has_todo": false,
    "can_replace": true
  },
  {
    "task_id": "PA19/Game.java",
    "compile_errors": 3,
    "test_result": [0, 0],
    "has_todo": true,
    "can_replace": false
  }
]
```

### 2.3 测试级别评估（Test-wise）

**定义**：为每个测试用例替换所有相关类

**优点**：
- 评估类间协作能力
- 更接近真实使用场景
- 结果更具区分度

**缺点**：
- 计算资源消耗大
- 需要依赖关系分析

**示例**：
```bash
python evaluation.py test-wise \
    --output output/result-PA19/gpt-3.5-turbo/result-full.json \
    --test datasets/testcase/test-PA19.jsonl \
    output/result-PA19/gpt-3.5-turbo/samples.jsonl
```

**输出格式**：
```json
{
  "PA19/GameTest": [
    {
      "test_id": "PA19/GameTest",
      "compilable": true,
      "n_pass": [10, 10],
      "has_todo": false,
      "can_replace": true
    }
  ]
}
```

### 2.4 项目级别评估（Project-wise）

**定义**：一次性替换所有生成的类

**优点**：
- 完整的端到端评估
- 最接近真实场景

**缺点**：
- 单个类的错误会影响整体
- 难以定位具体问题

## 三、评估指标

### 3.1 三级递进指标

| 指标 | 英文 | 计算方式 | 说明 |
|------|------|----------|------|
| **完成率** | Completion@k | 生成的类可替换的比率 | 是否生成了完整的类 |
| **编译率** | Compilation@k | 编译成功的比率 | 代码语法正确性 |
| **通过率** | Pass@k | 测试通过的比率 | 功能正确性 |

### 3.2 Pass@k 计算公式

论文中使用标准的 Pass@k 计算：

```
Pass@k = E[1 - C(n-c, k) / C(n, k)]
```

其中：
- `n`：样本总数
- `c`：通过的样本数
- `k`：取前 k 个样本

**实际计算代码**：
```python
from scipy.special import comb

def pass_at_k(n, c, k):
    """
    Calculate Pass@k metric
    
    Args:
        n: total number of samples
        c: number of correct samples
        k: number of samples to consider
    
    Returns:
        Pass@k score
    """
    if n - c < k:
        return 1.0
    return 1.0 - comb(n - c, k) / comb(n, k)

# 示例
n = 10  # 生成 10 个样本
c = 7   # 其中 7 个通过测试
k = 5   # 计算 Pass@5

score = pass_at_k(n, c, k)
print(f"Pass@5 = {score:.4f}")  # Pass@5 = 0.9833
```

### 3.3 指标解读

#### Completion@k（完成率）

**含义**：生成的代码是否完整

**检查项**：
- 是否包含完整的类定义
- 是否包含所有必需的方法
- 是否有 TODO 标记（有则说明未完成）

**影响因素**：
- LLM 的指令遵循能力
- 输出长度限制
- 代码理解能力

#### Compilation@k（编译率）

**含义**：生成的代码是否能编译通过

**常见编译错误**：
1. **类型错误**
   - 返回类型不匹配
   - 参数类型错误

2. **OOP 语义错误**
   - 错误实例化抽象类
   - 未实现接口方法
   - 访问权限错误

3. **语法错误**
   - 缺少分号
   - 括号不匹配

#### Pass@k（通过率）

**含义**：生成的代码是否通过测试用例

**常见测试失败原因**：
1. **功能错误**：实现逻辑不正确
2. **边界条件**：未处理特殊情况
3. **文档不符**：未遵循 Javadoc 说明
4. **Trivial 实现**：返回 null 或空实现

### 4.3 评估流程

```python
def evaluate_single_class(sample_file, output):
    """类级别评估流程"""
    samples = load_samples(sample_file)
    results = []
    
    for sample in samples:
        # 1. 创建测试环境
        test_env = TestEnv(...)
        
        # 2. 替换生成的代码
        test_env.replace(sample['target'], sample['completion'])
        
        # 3. 编译检查
        compile_errors = test_env.compile()
        
        # 4. 如果编译通过，运行测试
        if len(compile_errors) == 0:
            test_result = test_env.run_test(None)
        else:
            test_result = [0, 0]
        
        # 5. 记录结果
        results.append({
            'task_id': sample['task_id'],
            'compile_errors': len(compile_errors),
            'test_result': test_result,
            'has_todo': has_todo_in_code(sample['completion']),
            'can_replace': can_replace_class(sample['completion'])
        })
        
        # 6. 清理环境
        test_env.destory()
    
    save_results(results, output)
```

### 6.3 完成错误分析

**现象**：LLM 生成的代码仍包含 `// TODO` 标记

**原因**：
1. 输出长度限制
2. 指令遵循能力不足
3. 任务复杂度超出能力范围

**占比**：约 15% 的生成代码存在此问题

```bash
#!/bin/bash

PROJECTS=("PA19" "PA20" "PA21" "PA22")
MODEL="gpt-3.5-turbo"

for PROJECT in "${PROJECTS[@]}"; do
    echo "Evaluating $PROJECT..."
    
    # 1. 推理
    python inference.py \
        --model-path $MODEL \
        --num-sample 5 \
        --output output/result-$PROJECT/$MODEL/samples.jsonl \
        --data datasets/selective-context/data-$PROJECT.jsonl \
        --mode holistic
    
    # 2. 类级别评估
    python evaluation.py class-wise \
        --output output/result-$PROJECT/$MODEL/single_class.json \
        output/result-$PROJECT/$MODEL/samples.jsonl
    
    # 3. 测试级别评估

        --output output/result-$PROJECT/$MODEL/result-full.json \
        --test datasets/testcase/test-$PROJECT.jsonl \
        output/result-$PROJECT/$MODEL/samples.jsonl
done

echo "Evaluation complete!"
```

### 7.3 结果可视化

```python
import json
import matplotlib.pyplot as plt

def plot_results(results_file):
    """绘制评估结果"""
    with open(results_file, 'r') as f:
        results = json.load(f)
    
    # 提取数据
    tasks = [r['task_id'].split('/')[1] for r in results]
    compile_success = [1 if r['compile_errors'] == 0 else 0 for r in results]
    test_pass = [r['test_result'][0] / r['test_result'][1] if r['test_result'][1] > 0 else 0 for r in results]
    
    # 绘图
    fig, axes = plt.subplots(2, 1, figsize=(12, 8))
    
    # 编译成功率
    axes[0].bar(tasks, compile_success)
    axes[0].set_title('Compilation Success Rate')
    axes[0].set_ylabel('Success (0/1)')
    
    # 测试通过率
    axes[1].bar(tasks, test_pass)
    axes[1].set_title('Test Pass Rate')
    axes[1].set_ylabel('Pass Rate')
    axes[1].set_xlabel('Task')
    
    plt.tight_layout()
    plt.savefig('evaluation_results.png')

# 使用示例
plot_results('output/result-PA19/gpt-3.5-turbo/single_class.json')
```
