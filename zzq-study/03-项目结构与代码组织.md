## 二、项目命名规范

每个项目都有三个版本：

| 版本 | 命名 | 用途 | 说明 |
|------|------|------|------|
| 原始版 | PAxx | 完整项目 | 包含完整代码、测试、配置 |
| 上下文版 | PAxx-Context | 代码骨架 | 包含 TODO 标记，用于生成任务 |
| 解决方案版 | PAxx-Solution | 标准答案 | 包含完整实现，用于评估 |

在 projects/ 里，每个 PA 项目基本都有三份（以 PA19 为例）：

PAxx/（我更愿意叫 TODO 版 / 任务版）

用途：作为“要让模型补全的目标代码”。
特征：类/注释/导入都很完整，但关键实现处会有 // TODO。
例子：projects/PA19/.../Cell.java 里 fromChar 有 // TODO。

PAxx-Context/（Context 版 / 依赖签名版）

用途：给模型提供“上下文信息”的素材库（主要是依赖类的结构与方法签名）。
特征：很多方法体是空的，只保留类结构/字段/方法签名（非常像 interface stub）。
例子：projects/PA19-Context/.../Cell.java 里构造器和 fromChar 都是空实现，仅保留签名。
PAxx-Solution/（Solution 版 / 标准答案版）

用途：评估时的参考实现/正确答案，用于编译、跑测试、以及“替换对照”。
特征：完整可运行实现，通常还更严谨（比如带 @NotNull/@Nullable 等）。
一句话：

PAxx = 题目（带 TODO）
PAxx-Context = 给题目配的上下文壳子（签名为主）
PAxx-Solution = 标准答案（评测用）

## 四、四个项目详细对比

### 4.1 PA19 - 管道游戏（Pipe Game）

**项目类型**：图形界面游戏  
**主要功能**：玩家通过放置管道，将水流从源头引导到终点

**核心类结构**：
```
game/
├── Game.java              # 游戏主逻辑
├── Map.java               # 游戏地图
├── CellStack.java         # 撤销栈
├── DelayBar.java          # 延迟条
├── PipeQueue.java         # 管道队列
├── map/
│   └── cells/
│       ├── Cell.java           # 单元格基类（抽象类）
│       ├── FillableCell.java   # 可填充单元格
│       ├── TerminationCell.java # 终止单元格（源头/终点）
│       └── Wall.java           # 墙壁
└── pipes/
    └── Pipe.java           # 管道
```

**PA19特点**：
- 抽象类：Cell（单元格基类）
- 继承：FillableCell、TerminationCell、Wall 继承 Cell
- 封装：private 字段 + public 方法
- 接口：MapElement 接口
- 枚举：Direction、Pipe.Shape

### 4.2 PA20 - JesonMor 棋类游戏

**项目类型**：棋类游戏  
**主要功能**：两人对弈的棋类游戏，包含完整的游戏规则和 AI 对手

**核心类结构**：
```
assignment/
├── Main.java              # 主程序
├── JesonMor.java          # 游戏实现
├── piece/                 # 棋子规则
│   ├── Rule.java          # 规则接口
│   ├── Knight.java        # 骑士棋子
│   ├── Archer.java        # 弓箭手棋子
│   └── [各种规则类]
├── player/                # 玩家
│   ├── Player.java        # 玩家抽象类
│   ├── RandomPlayer.java  # 随机玩家
│   └── ConsolePlayer.java # 控制台玩家
└── protocol/              # 游戏协议
    ├── Game.java          # 游戏抽象类
    ├── Configuration.java # 游戏配置
    ├── Piece.java         # 棋子抽象类
    ├── Move.java          # 移动
    └── Color.java         # 颜色枚举
```

**PA20特点**：
- 抽象类：Game、Piece、Player
- 继承：多种棋子继承 Piece
- 多态：不同 Player 实现（RandomPlayer、ConsolePlayer）
- 接口：Rule 接口
- 设计模式：策略模式（不同的移动规则）

### 4.3 PA21 - Inertia 惯性游戏

**项目类型**：文本冒险游戏  
**主要功能**：玩家在网格中移动，收集宝石，躲避地雷

**核心类结构**：
```
pa1/
├── Main.java              # 主程序
├── InertiaTextGame.java   # 文本游戏
├── controller/            # 控制器
│   ├── GameController.java
│   └── GameBoardController.java
├── model/                 # 模型（实体类）
│   ├── BoardElement.java  # 棋盘元素接口
│   ├── Cell.java          # 单元格
│   ├── Entity.java        # 实体抽象类
│   ├── Player.java        # 玩家
│   ├── Gem.java           # 宝石
│   ├── Mine.java          # 地雷
│   ├── Wall.java          # 墙壁
│   ├── GameBoard.java     # 游戏棋盘
│   └── GameState.java     # 游戏状态
├── view/                  # 视图
│   ├── GameView.java
│   └── GameBoardView.java

    └── GameStateSerializer.java
```

**P21特点**：
- MVC 架构：Model-View-Controller
- 抽象类：Entity
- 接口：BoardElement
- 继承：Player、Gem、Mine、Wall 继承 Entity
- 封装：完善的 getter/setter

## 七、项目间关系

四个项目递进，难度和复杂度逐步提升：

```
PA19 (2019) → PA20 (2020) → PA21 (2021) → PA22 (2022)
  ↓             ↓             ↓             ↓
简单游戏     棋类游戏      文本游戏      推箱子游戏
单一职责     多角色对战    MVC架构       多设计模式
```

## 八、使用建议

### 8.1 学习路径

推荐按以下顺序使用：
1. **PA19**：理解基本的类结构和游戏逻辑
2. **PA20**：学习抽象类、接口和多态
3. **PA21**：掌握 MVC 架构和复杂对象关系
4. **PA22**：学习设计模式和高级 OOP 特性

### 8.2 LLM 评估路径

建议从简单到复杂：
1. 先在 PA19 上测试基本能力
2. 逐步增加难度到 PA22
3. 对比不同项目的表现差异

### 8.3 数据集选择

- **入门研究**：PA19（最简单）
- **中等难度**：PA20、PA21
- **高难度挑战**：PA22（最复杂）
