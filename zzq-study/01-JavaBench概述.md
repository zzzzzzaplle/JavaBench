# JavaBench 概述

## 一、研究背景

JavaBench 是首个专注于面向对象编程（OOP）特性的项目级 Java 基准测试，旨在评估大语言模型（LLM）在生成复杂 Java 代码方面的能力。

### 1.1 现有基准测试的三大不平衡

1. **编程语言不平衡**
   - 95.8% 的基准测试涉及 Python
   - Java 作为第二大流行语言，相关基准测试严重不足

2. **代码粒度不平衡**
   - 超过 83.3% 的基准测试是函数/语句级别
   - 仅有极少数扩展到类/项目级别，且仅限于 Python

3. **缺乏高级特性**
   - 现有基准主要评估基本编码技能
   - 忽略了面向对象编程（OOP）的核心特性：封装、继承、多态

## 二、JavaBench 核心特点

### 2.1 基本构成

| 指标 | 数值 |
|------|------|
| 项目数量 | 4 个 |
| Java 类总数 | 106 个 |
| 方法总数 | 389 个 |
| 测试用例数 | 396 个 |
| 平均代码行数 | 1740 行/项目 |
| 测试覆盖率 | 92% |

### 2.2 质量保证

- 由 282 名本科生验证
- 平均测试通过率：90.93/100
- 包含 Mocking 等高级测试技术

### 2.3 四个项目简介

| 项目 | 年份 | 项目类型 | 主要特性 |
|------|------|----------|----------|
| PA19 | 2019 | 管道游戏 | 图形界面游戏，涉及地图、管道、游戏逻辑 |
| PA20 | 2020 | 棋类游戏 | JesonMor 棋类游戏，包含完整的游戏规则 |
| PA21 | 2021 | 惯性游戏 | Inertia 文本游戏，包含复杂的游戏状态管理 |
| PA22 | 2022 | 推箱子游戏 | Sokoban 游戏，包含完整的游戏引擎 |

## 三、研究贡献

### 3.1 重要性
- 首个专注于 OOP 特性的项目级 Java 基准测试
- 有助于深入理解 LLM 在生成复杂 Java 代码方面的优势与局限

### 3.2 新颖性
- 引入了一套系统的、多层次的评估设计
- 为未来的项目级代码生成评估提供了参考

### 3.3 评估与洞察
- 揭示了 LLM 在当前阶段处理 Java OOP 项目的真实能力水平
- 指出了最优的上下文设置
- 错误案例分析为未来改进指明了方向

## 四、主要发现

### 4.1 LLM 能力现状
- 所有被研究的 LLM 均无法完整正确地生成任何一个项目
- 最佳表现（Pass@5）仅为 41.17%，远低于本科生的 90.93%

### 4.2 上下文设置影响
- **选定上下文（最佳）**：仅包含方法签名的上下文
- **最大上下文**：整个项目骨架（信息过多）
- **最小上下文**：仅当前类（信息过少）

### 4.3 合成策略影响
- **整体合成**（推荐）：一次性生成一个类的所有方法
- **独立合成**：逐个方法独立生成
- **增量合成**：基于已生成方法逐步完善

### 4.4 LLM 常见错误
1. 完成错误：忽略待完成代码（TODO）
2. 编译错误：不理解 OOP 语义
   - 错误实例化抽象类
   - 违反封装原则
3. 测试失败：未遵循文档或提供 trivial 实现
